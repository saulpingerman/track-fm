## 5. Cross-cutting patterns worth institutionalizing

**Copy-forward hygiene.** *Doctrine:* every mechanism-level fix
implicitly claims scope; scope must be enumerated. If you patch a
`mkdir(exist_ok=True)`, you own the audit of every other
`mkdir(exist_ok=True)` in the tree; if you fix a chain script's
`>`-truncate, you own every `run()` copy elsewhere. *Mechanism:* fix
commits must include a `grep -n <pattern>` proof in the message; a
pre-commit hook re-runs the grep and blocks the commit if new
occurrences exist post-patch. Apply retroactively to F1/F5/F7/F10/F20/F27/F30 tomorrow.

**Checkpoint-vs-config contract.** *Doctrine:* a checkpoint is a
runnable object only when its config, its raster, and its RNG state
travel with it. A `state["config"]` that no loader ever asserts against
is worse than no config at all, because it lies about
reproducibility. *Mechanism:* on save, write `state["config_sha"]`,
`state["raster_meta"]` (shape, sha256, channel_names), `state["rng"]`
(torch/cuda/numpy/python); on load, assert the passed cfg's relevant
subtree hashes equal to `state["config_sha"]`, error on mismatch by
default with `--allow-config-drift` opt-in for informed rescoring.

**RNG hygiene.** *Doctrine:* "seed = 17" without CUDA/numpy/python/
worker seeding is not reproducibility; it is aesthetics. Every headline
gain in a scaling paper needs a σ; σ needs replicates; replicates need
determinism inside a seed AND controlled variance across seeds.
*Mechanism:* one `seed_everything(seed)` helper that hits
`torch.manual_seed`, `torch.cuda.manual_seed_all`, `np.random.seed`,
`random.seed`, `cudnn.deterministic`, TF32 pin, and returns a
`worker_init_fn`; called once at process start; all headline yamls run
{17, 18, 19}; scaling curves report band-of-seed rather than
point-of-seed.

**On-disk poisoned JSON hygiene.** *Doctrine:* any JSON that names
itself as "the headline" for a decision (`c1_traffic.json`, saved eval
JSONs from `xgeom_matrix.py`, tune_kalman outputs) is a public API of
the repo; when the pipeline changes, the JSON is not automatically
invalidated. Silent staleness poisons future readers, PRs, and paper
tables. *Mechanism:* every headline JSON carries a
`{"metrics_version": "v2", "code_sha": ..., "produced_by": ...}` header;
readers assert version; a `make audit-jsons` walks the repo and lists
JSONs that do not match the live version. Delete or regenerate on
mismatch.
