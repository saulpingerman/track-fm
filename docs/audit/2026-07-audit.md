14 specialist audit lenses (loss math, metric math, leakage, numerics,

## Verdict

The training stack and model wiring are fundamentally sound (the causal
mask, cone normalization chain, loader sharding, and context wiring all
came back clean under hostile lenses). The defects concentrate in TWO
places: (1) the EVALUATION layer — grid/lattice conventions, baseline
loss floors, metric definitions — where several small biases have been
silently shaping numbers; and (2) EXPERIMENT PROCESS — test-split usage,
statistics of the headroom gates, chain-script robustness. Nothing found
invalidates the big directional conclusions (cone vs fixed, capacity
scaling, traffic > weather), but several exact numbers need re-scoring
with fixed code, and the gates' effect sizes carry more uncertainty than
reported.

## CONFIRMED findings (3-lens verified)

### F1 [critical] `scripts/xgeom_matrix.py:20`

Test split repeatedly consumed for model/architecture selection: xgeom_matrix.py defaults SPLIT='test' (head_geom_matrix_queue.sh passes 'test' explicitly), eval/forecast.py:35 and cli.py:157 default split='test', and campaign.sh/campaign_gated.sh pass no --split — so the scaling-series eval JSONs, the head ablation, and the 2x2 geometry-x-head matrix were all scored on test, and the cone-geometry and Fourier-vs-direct decisions were made on test numbers.

**Failure scenario:** Paper 'held-out test' numbers on v1 test (2024-12-09..2025-02-26) are selection-biased: cone+fourier was chosen because it won on this exact split, and since the corpus ends 2025-02 there is no untouched later window left for a clean final eval. Any reviewer checking split hygiene finds the same eval JSONs.

**Fix:** Flip defaults in xgeom_matrix.py, eval/forecast.py, eval/xgeometry.py, cli.py to split='val'; make split a required arg for anything writing decision JSONs; log every test-split touch. For the paper, either declare v1 test a development split and carve a genuinely untouched final window (extend data to 2025-03+ or re-split), or report test numbers with an explicit selection-on-test caveat.

### F2 [critical] `src/trackfm/training/losses.py:72`

Gaussian-baseline exp-then-log(density+1e-10) floor (also cloned in eval/baselines.py:46 and eval/search.py:121) caps baseline penalty on bad misses at ~23 nats and creates rank-tie plateaus: baseline losses are flattered, tune_kalman/fit_sigma_per_horizon selection surfaces are corrupted (small-sigma configs scored on the artifact; r=1e-5 grid boundary chosen in all three saved evals), and search_ranks' strict '>' gives badly-missed baselines absurdly good ranks (dr_tuned/kalman rank 65/4096 for ANY miss 0.14–0.75 deg; rank 1 when the whole grid underflows).

**Failure scenario:** Verified numerically (G=64, grid_range=0.3): DR miss at 0.53 deg reports loss 22.8 vs true 117.4; with fitted sigma 0.005 the floor binds beyond ~0.03 deg. Paper Table-1 dr/lp/kalman losses and the 'Nx lower loss than DR' ratios are computed against flattered baselines; stratified hard-decile model-vs-tuned-baseline comparisons and capture@k are corrupted pro-baseline and can flip conclusions to false negatives. Model-side losses (log_softmax) are unaffected — this poisons only the honesty apparatus.

**Fix:** Compute baseline grid log-densities analytically in log space: logq = -0.5*(zx^2+zy^2) - logsumexp(...); no epsilon, no underflow, exact tail ordering. Use logq in the CE and rank on logq; optionally break rank ties conservatively. Re-run fit_sigma_per_horizon, tune_kalman, forecast.py tables, and stratified deciles for affected checkpoints.

### F3 [critical] `src/trackfm/eval/xgeometry.py:100`

ranks_on_fine_grid resamples with grid_sample(align_corners=False) while the density lattice is linspace(-R,R,G) — the align_corners=True convention used everywhere else (loss grid, Fourier basis, crops.py) — so the resampled field is compressed by (G-1)/G and misregistered by up to a native cell toward the canvas edge.

**Failure scenario:** Verified empirically: a density peaked exactly at native node (55,55) with the target at that physical location gets fine-grid rank 13 instead of 1 on a 2h cone canvas; the peak shifts 1.2 km inward. The shift scales with native cell size (~4.4 km cone at 2h vs ~1.06 km fixed), so cone runs are penalized ~4x harder in val_km2_at_capture90 / val_fixcell_p90rank / val_fixgrid_p90rank and score_geometry tables — a systematic anti-cone bias in exactly the flagship cone-vs-fixed comparison.

**Fix:** Pass align_corners=True to the grid_sample call (fine-cell centers and truth binning are mutually consistent and need no change); update tests/eval/test_xgeometry.py::test_truth_cell_is_argmax_when_density_is_peaked_there which pinned the wrong convention; re-score cone/fixed comparisons and re-run xgeom_matrix before concluding geometry ordering.

### F4 [critical] `scripts/build_traffic_prior.py:24`

Traffic-prior leakage guard (TRAIN_END=2024-09-21) is v1-specific and silently fails for v3sub200 — the dataset being materialized right now for conditioning experiments spans the first 200 clean days (2023-01..2023-07), so its val/test days fall inside the prior's aggregation window AND inside v1's train period.

**Failure scenario:** A geo_traffic-conditioned model trained/evaluated on v3sub200 gets a conditioning channel built partly from the very val/test days it is scored on, inflating the conditioning-headroom conclusion; worse, warm-starting from any v1 checkpoint (the zero-init design is explicitly warm-start friendly) scores the model on days it pretrained on (v3sub200 val/test are v1 TRAIN days).

**Fix:** Materialize v3 subsets from the end of the v1 train period backwards (or set start_date so v3 val/test fall after 2024-09-21); add a hard assertion when loading traffic_prior.npz that the eval split's first day > stored train_end; extend the mandatory MANIFEST split-overlap check to cover context rasters and warm-start lineage.

### F5 [major] `src/trackfm/eval/xgeometry.py:170`

score_geometry computes 'fixcell' ranks WITHOUT restrict_deg=0.3 yet comments and the xgeom_matrix table claim direct comparability to fixed-grid numbers; the correct restricted call exists in pretrain.py:198-200 but is never used here.

**Failure scenario:** For cone checkpoints at 2h (R=1.25 deg) the unrestricted fixcell grid spans ~17x more area (267x267 vs 64x64 cells) and includes the ~20% fastest/hardest targets that the fixed grid censors, so cone reads worse than apples-to-apples — the 2x2 matrix can pick the wrong geometry.

**Fix:** Add a third fine-grid call in score_geometry with cell_deg=FIXED_NATIVE_CELL_DEG, restrict_deg=0.3 (mirroring pretrain.py's val_fixgrid metric) as the comparable column; relabel existing unrestricted fixcell numbers as population-unmatched.

### F6 [major] `src/trackfm/training/pretrain.py:230`

val_km2_at_capture90 is the conditional 0.9-quantile over captured (on-canvas) targets only, ignoring censored targets — unlike score_geometry's same-named metric, which counts censored samples in the denominator and returns None when ceiling < 0.9.

**Failure scenario:** Fixed-grid run at 2h: ~20-25% of targets escape the ±0.3 deg canvas and are dropped, so quantile(0.9) of the survivors is really km2-to-capture-~70%-of-all, yet logged as capture90 — while cone runs (ceiling ~1.0) report a true 90%. MLflow curves flatter fixed at long horizons in the flagship comparison, and the two eval tools silently disagree on the same checkpoints.

**Fix:** Compute the 0.9 point over all available samples with censored treated as +inf: k = ceil(0.9*n_avail); log sorted(fine_valid)[k-1] only if len(fine_valid) >= k, else NaN (val_ceiling gives the companion signal).

### F7 [major] `src/trackfm/eval/search.py:46`

search_ranks floor-indexes truth assuming edge-tiled cells of width 2R/G, but density values live at linspace(-R,R,G) nodes with spacing 2R/(G-1): 24.6% of targets (verified by simulation at G=64) are assigned a non-nearest lattice point, always shifted away from center; FIXED_NATIVE_CELL_DEG also understates the interior lattice pitch area by 3.2%.

**Failure scenario:** With sigma=0.003 deg the trained density is sub-cell peaked, so a well-predicted target in the disagreement region has p_truth read one cell off-peak (density down ~e^-5), inflating rank from ~1 to ~5-20. Native metrics (native_k90_cells, native_km2_to_capture90, capture@1/3/10, historical val_p90rank) are systematically pessimistic for models and baselines alike; absolute containment numbers are wrong even though ratios largely survive.

**Fix:** Index truth by nearest lattice point: idx = ((targets + grid_range)/(2*grid_range)*(G-1)).round().long().clamp(0,G-1), and use cell = 2*grid_range/(G-1) for km2 conversion (eval-side fix; migrating to a cell-center lattice would invalidate trained checkpoints).

## Top recommendations

1. Metric fixes (F3/F5/F6/F7) land together as one "metrics v2" commit;
   every cross-run table after this date is rescored with metrics v2 —
   never mix versions in one table.
2. Test split is retired from all selection; harness defaults flip to
   val (F1). Test appears once, in the paper-final table.
3. Baselines get analytic log-densities, retuned sigma (F2).
4. Every gate that motivates a build decision gets: underway filter,
   track-level block bootstrap CIs, and a held-out replication day-set.
5. Chain scripts: append logs, export allocator conf everywhere, arm-
   once handoff locks (F19/F20/F21).
6. Add resume support (optimizer state in last.pt) before any run
   longer than the xlarge pair (F22).
7. materialize pass-1 resume: verify no silently-dropped buffered rows
   (F9) before trusting v3sub200; re-run affected days if found.

   channel at the next materialization rev (v3 already carries heading).
